IBM watsonx Governance Bridge

A Python request-time governance edge that sits in front of watsonx.ai and enforces a buyer's AI Procurement Decision Card on every chat or text-generation call. Allow forwards; deny returns 403 with the rationale; require_approval returns 409. Every decision emits a tool_invocation_* event to the same audit-stream spine as the rest of the Kinetic Gain portfolio.

Request flow

The bridge derives one or more tool_names from each request — model, deployment, operation family, and any declared function-calling tools — then evaluates them against deny-trumps-allow PolicyBundles before any upstream call. No tokens are generated until the decision is allow.

allow → upstream forward deny → 403 require_approval → 409

What gets checked

tool_name derivation

For each request the bridge derives one or more tool_names from the route + declared tools[] array:

  • watsonx.model.<model_id>
  • watsonx.deployment.<deployment_id>
  • watsonx.operation.text.chat
  • watsonx.operation.text.generation
  • tool.<name> per declared tool

How rules resolve

Deny-trumps-allow

Each rule has a priority, an effect (allow · deny · require_approval), a tool/caller regex, and an optional when.expr Python expression evaluated against the request context. The highest-priority matching rule wins; ties break to deny.

Default outcome when no rule matches is configurable via DEFAULT_OUTCOME (defaults to deny).

Audit spine

tool_invocation events

Every decision — allow, deny, or require_approval — emits a tool_invocation_* event to audit-stream-py. The watsonx path writes to the same tamper-evident spine as the Azure OpenAI sibling and the MCP request gate.

Each decision carries the matched rule IDs and the source Decision Card URLs back to the caller.

Auth + deploy

IBM Cloud IAM, Code Engine

IBM Cloud IAM auth via pre-minted token (IBM_CLOUD_IAM_TOKEN) or API key minted at startup (IBM_CLOUD_API_KEY). Bridge is read-only — no tenant prompts or tokens land on disk.

Deploy manifest in infra/ ships as IBM Code Engine; any HTTP container runtime works.

Routes

All routes accept JSON. Required request headers: x-kg-caller-id and x-kg-environment. The bridge stamps an x-kg-correlation-id on every response for audit-stream cross-reference.

Usage

Three real shapes from the repo — switch with the tabs below. No SDK install for the curl path; the Python path uses the in-process Broker directly.

from ibm_watsonx_governance_bridge import PolicyBundle, Broker
from ibm_watsonx_governance_bridge.models import PermissionRequest

broker = Broker(default_outcome="deny")
broker.add_bundle(PolicyBundle(
    bundle_id="enterprise-watsonx-prod-2026",
    decision_card_url="https://enterprise.example/.well-known/decisions/DEC-200.json",
    rules=[ ... ]  # see PolicyBundle tab
))

decision = broker.check(PermissionRequest(
    caller_id="claims-assistant-prod",
    tool_name="watsonx.model.ibm/granite-3-8b-instruct",
    context={"environment": "production"},
))

# decision.outcome           -> "allow" | "deny" | "require_approval"
# decision.matched_rules     -> ["allow-approved-foundation-models", ...]
# decision.decision_card_refs -> ["https://enterprise.example/.well-known/decisions/DEC-200.json"]
curl -X POST http://localhost:8080/api/governed/models/ibm/granite-3-8b-instruct/text/chat \
  -H 'x-kg-caller-id: claims-assistant-prod' \
  -H 'x-kg-environment: production' \
  -H 'content-type: application/json' \
  -d '{"project_id":"proj-123","messages":[{"role":"user","content":"Summarize the policy"}]}'

# allow            -> 200 + upstream watsonx.ai response verbatim + x-kg-correlation-id
# deny             -> 403 + rationale + offending Decision Card URL
# require_approval -> 409 + matched rule + Decision Card URL
{

Posture

Reference scaffolding for runtime AI governance on the watsonx.ai data path. The bridge gates request shape against PolicyBundles derived from buyer Decision Cards; it does not by itself establish compliance with HIPAA, the EU AI Act, NIST AI RMF, or any other regulatory regime. Production deployments still need their own attestation, data-protection review, and IAM hardening.

Per the standing Suite public-language guardrail: readiness · evidence · posture · controls · scaffolding — never "compliant" or "certified" without an external attestation specific to the deployed environment.
